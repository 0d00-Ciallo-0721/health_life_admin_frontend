更新时间：2026-04-10

适用前端项目：`health_life_admin_frontend`

后端管理接口根路径：

```text
http://127.0.0.1:8000/api/admin/v1
```

前端项目当前统一请求入口见：
- `assets/js/api.js`

## 1. 鉴权说明

### 1.1 登录接口

- 方法：`POST`
- 路径：`/auth/login/`
- 是否需要登录：否

请求体：

```json
{
  "username": "admin",
  "password": "123456"
}
```

成功响应：

```json
{
  "user_info": {
    "id": 1,
    "username": "admin",
    "nickname": "管理员",
    "avatar": "",
    "roles": ["super_admin"]
  },
  "access_token": "xxx",
  "token": "xxx",
  "refresh_token": "xxx"
}
```

前端注意：
- 登录接口是后台管理系统里少数“**不走 `{code,msg,data}` 包装**”的接口。
- 登录后统一使用 `access_token`。
- 所有后续请求在 Header 中带：

```http
Authorization: Bearer <access_token>
```

### 1.2 常见鉴权错误

- `401`：未登录或 token 失效
- `403`：已登录，但该后台账号没有对应权限

---

## 2. 通用响应规范

除登录接口外，后台接口统一由自定义 Renderer 包装，标准格式如下：

```json
{
  "code": 200,
  "msg": "success",
  "data": {}
}
```

说明：
- 列表接口通常读 `res.data`
- Mongo 列表类接口通常额外返回 `pagination`
- 部分风控接口返回 `data.list`

前端建议统一处理：

```js
const list = res?.data?.list || res?.data || res?.results || []
```

## 4. 详细接口说明

## 4.1 仪表盘

### 4.1.1 获取首页概览

- 方法：`GET`
- 路径：`/dashboard/summary/`

响应：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "cards": [
      {
        "title": "总用户数",
        "value": 100,
        "today_added": 3,
        "icon": "User"
      }
    ],
    "trend": [
      {
        "date": "04-10",
        "value": 22
      }
    ],
    "chart_placeholder": false
  }
}
```

前端说明：
- `cards` 用于首页卡片
- `trend` 用于近 7 日趋势图

### 4.1.2 获取饮水/日志宏观统计

- 方法：`GET`
- 路径：`/business/stats/journal/`

响应核心字段：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "today": "2026-04-10",
    "water_stats": {
      "total_users_logged": 12,
      "completed_users": 9,
      "completion_rate_pct": 75.0,
      "total_ml_drank": 22500,
      "total_cups_drank": 90.0
    },
    "intake_stats": {
      "total_users_logged": 8,
      "total_records": 26,
      "avg_records_per_user": 3.3
    }
  }

```

前端说明：
- 如果页面还在按“杯数”展示，可以直接用 `total_cups_drank`
- 如果页面要展示更准确单位，优先使用 `total_ml_drank`

## 4.3 角色管理

### 4.3.1 角色字段说明

后端对前端开放字段：
- `id`
- `name`
- `description`
- `menus`
- `created_at`

后端内部也会返回：
- `role_name`
- `role_key`

前端说明：
- 页面只需要使用 `name` 和 `description`
- 后端会自动把 `description/name` 映射生成 `role_key`
- `menus` 传菜单 ID 数组即可

### 4.3.2 创建角色

- 方法：`POST`
- 路径：`/system/roles/`

请求体：

```json
{
  "name": "内容审核员",
  "description": "content_reviewer",
  "menus": [1, 2, 5]
}
```

响应示例：

```json
{
  "code": 201,
  "msg": "success",
  "data": {
    "id": 3,
    "name": "内容审核员",
    "description": "content_reviewer",
    "role_name": "内容审核员",
    "role_key": "content_reviewer",
    "menus": [1, 2, 5],
    "created_at": "2026-04-10T10:00:00+08:00"
  }
}
```

前端说明：
- 创建成功的 HTTP 状态码可能是 `201`
- 读取数据时统一从 `data` 取即可

## 5. 前端联调建议

### 5.1 推荐的读取方式

建议统一：

```js
const payload = await API.someRequest()
const data = payload.data
```

列表建议：

```js
const list = payload?.data?.list || payload?.data || []
const pagination = payload?.pagination || payload?.data?.pagination || null
```

### 5.2 重点注意的兼容字段

- 登录：读 `access_token`
- 角色：提交 `name`、`description`，不要自己拼 `role_key`
- 通知：提交 `type: 'public'` 即可
- 日志：页面使用 `request_path`
- 商家：页面使用 `avg_price`
- 挑战：页面使用 `description`，后端自动映射 `desc`
- 补救方案：页面使用 `description` 和 `points_cost`

### 5.3 仍建议前端保留的兜底

虽然当前后端已经基本统一，前端请求工具仍建议继续保留下面的兼容读取：

```js
const list = res?.data?.list || res?.data || res?.results || []
```

原因：
- 管理后台有普通列表
- 有 Mongo 分页列表
- 也有风控接口这种 `data.list` 结构
